  最关键的一个抽象： AgentRuntime
  是整个项目的"上帝对象"——它不直接做任何事，而是把所有能力（actions、providers、
  services、evaluators、memory）都注册进来，然后通过三阶段
  pipeline（message-handler → planner-loop → evaluator）按需调用。这使得 Eliza
  可以用同一套运行时支持完全不同的 agent persona。

  工程上的巧思： 消息处理被明确分成三个独立文件，每一跳的职责边界极其清晰——Stage
   1 只决定"该怎么处理"，Stage 2 只负责"多轮工具调用循环"，Stage 3
  只做"输出验证和副作用"。这种"管道式职责隔离"让每个环节可以独立测试、独立替换，
  也是新手最值得学习的设计模式：不要把决策、执行、验证写在同一个函数里。

  Step 2 — 关键路径追踪（信号 → 决策 → Self-Improving）

  === 主路径 ===

  1. [HTTP 信号接收] (异步)
     → /packages/agent/src/api/agent-routes.ts: handleAgentMessage()
     - 解析 POST /api/agents/:id/message body
     - 构造 Memory 对象

  2. [构建上下文 State]
     → /packages/core/src/runtime.ts: composeState()
     - 遍历所有已注册 Providers，并行调用 provider.get()
     - 合并成 State 对象（含历史 memories、当前 room 信息）

  3. [Stage 1 路由判断]
     → /packages/core/src/runtime/message-handler.ts:
  routeMessageHandlerOutput()
     - 调用 useModel() 做首轮 LLM 判断
     - 返回 MessageHandlerRoute（ignored / simple_reply / planning_needed）

  4. [Stage 2 规划循环] (关键跳，多轮工具调用)
     → /packages/core/src/runtime/planner-loop.ts: runPlannerLoop()
     - 内部调用 buildActionCatalog() 获取可用 actions
     - 调用 validated-model-call.ts: validateModelCall() 执行 LLM
     - 解析工具调用: response-grammar.ts: parseToolCallsFromResponse()
     - 执行每个工具: execute-planned-tool-call.ts: executeToolCall()
     - 记录轨迹: trajectory-recorder.ts: recordTrajectoryStep()
     - 循环直到无工具调用

  5. [链上 / 数据库 / 文件系统交互] (在 executeToolCall 内)
     → /packages/core/src/runtime/execute-planned-tool-call.ts:
  executeToolCall()
     - action.handler(runtime, message, state) 调用具体插件
     - 例如 plugin-wallet 执行链上交易，plugin-sql 写 DB

  6. [Stage 3 输出验证]
     → /packages/core/src/runtime/evaluator.ts: runEvaluator()
     - 验证 planner 输出符合预期
     - applyEvaluatorEffects() 执行后置副作用

  7. [执行结果持久化]
     → /packages/core/src/runtime.ts: createMemory()
     - 内容 secret 脱敏后写入 DatabaseAdapter
     - 触发 applyPipelineHooks("after_memory_persisted")

  8. [Self-Improving：记忆检索增强下一轮]
     → /packages/core/src/runtime.ts: searchMemories()
     - 下一轮 composeState() 时向量搜索历史轨迹
     - BM25 rerank: search.ts: BM25.search()
     - 相关历史注入新轮 planner 上下文 → 形成闭环

  === 异常分支 ===

  3a. [Stage 1 → ignored]
      → message-handler.ts: routeMessageHandlerOutput() 返回 "ignored"
      → 直接 return，不进入 planner

  4a. [工具调用失败]
      → execute-planned-tool-call.ts: executeToolCall() 抛异常
      → planner-loop.ts 捕获，记录错误到 trajectory
      → LLM 拿到错误信息重新规划（内置 retry 逻辑）

  4b. [达到最大轮次]
      → planner-loop.ts: ChainingLoopConfig.maxIterations 触发
      → 强制退出循环，返回当前 trajectory

  最关键的一跳： runPlannerLoop() → /packages/core/src/runtime/planner-loop.ts

  原因： 这是"自主性"真正发生的地方——LLM 在这里进行多轮工具选择和执行，决定是否
  下单、如何调用链上操作。整个系统的智能性（决策质量、工具编排、错误恢复）都在这
   1800 行里体现。Self-improving 闭环也通过 trajectory 记录 → createMemory()
  持久化 → 下轮 searchMemories() 检索历史来实现。

  Step 4 — 3 个可借鉴到 AutoArb 的设计点

  设计点 1：三阶段管道 + 职责隔离

  解决什么问题： Agent
  决策逻辑越写越大，最后变成一个几千行的"上帝函数"，impossible to test 或
  replace 某一环节。

  怎么落地到 AutoArb：
  signal-handler.ts   → 只做信号路由（是否值得分析？）
  analysis-loop.ts    → RAG 查询 + LLM 决策 + Polymarket 下单
  trade-auditor.ts    → 验证成交状态 + 更新仓位记录
  每层独立可测，每层可被替换（如换 LLM）而不影响其他层。

  代码位置：
  /packages/core/src/runtime/message-handler.ts (Stage 1) → planner-loop.ts
  (Stage 2) → evaluator.ts (Stage 3)

  ---
  设计点 2：DatabaseAdapter 抽象层 + 批量优先接口

  解决什么问题： 早期用 SQLite，后来要换 PostgreSQL 或向量
  DB，改动牵一发动全身。

  怎么落地到 AutoArb：
  // 定义一次抽象接口
  interface ITradeStore {
    savePositions(positions: Position[]): Promise<UUID[]>   // 批量写
    queryPositions(filters: PositionFilter): Promise<Position[]>
    searchSimilarTrades(embedding: number[]): Promise<Position[]>  // 向量检索
  }
  // 实现1: SqliteTradeStore（开发用）
  // 实现2: PostgresTradeStore（生产用）
  // 实现3: InMemoryTradeStore（测试用）
  切换存储层零改业务代码。

  代码位置：
  /packages/core/src/database.ts: DatabaseAdapter<DB> —
  注意其方法全是批量签名（createMemories 不是 createMemory），这是为了减少 IO
  轮次。

  ---
  设计点 3：createMemory 是唯一的 Secret 脱敏点

  解决什么问题： Agent 运行中随时可能把私钥、API key 写入 log 或
  DB，分散在各处的脱敏逻辑容易漏掉一处就造成泄漏。

  怎么落地到 AutoArb：
  // 所有写入 DB/日志的操作必须经过这一个函数
  async function saveTradeRecord(record: TradeRecord): Promise<UUID> {
    const cleaned = redactSecrets(record, getRegisteredSecrets())
    // 只有这里能写 DB
    return db.insert(trades).values(cleaned)
  }
  把"脱敏"和"持久化"强制绑定在同一个函数里，而不是各处手动调用 redact。

  代码位置：
  /packages/core/src/runtime.ts: createMemory() line 8287 — 关键：脱敏发生在写
  DB 之前，且封装了 this.getSecretsForRedaction() 统一管理所有已注册 secret。
